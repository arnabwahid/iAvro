Here’s what “bump CACHE_VERSION” means and why/when to do it.

What it is: a manual version tag used in cache keys in .github/workflows/ci.yml. Changing it (e.g., v1 → v2) makes all existing caches “invisible” so CI rebuilds fresh.
Where it’s used:
Pods cache key: includes ${{ runner.os }}-${{ env.CACHE_VERSION }}-pods-${{ hashFiles('Podfile.lock') }}.
DerivedData cache key: includes ${{ runner.os }}-${{ env.CACHE_VERSION }}-derived-${{ hashFiles('AvroKeyboard.xcodeproj/project.pbxproj','Podfile.lock') }}-${{ env.SCHEME }}.
Why bump:
Cache looks stale or corrupted (strange build/link errors that disappear locally).
Tooling changed significantly (new Xcode, macOS runner, large dependency rewire) and you want a clean slate.
Debugging cache behavior (force a guaranteed miss to confirm caching is the issue).
What it does:
Forces a full pod install (from CDN) and a clean Xcode build in the next run.
Repopulates new cache entries keyed with the new version.
How to do it:
Edit CACHE_VERSION in .github/workflows/ci.yml from v1 to v2 (or v3, etc.).
Commit and push; the next CI run downloads nothing from the old caches and re-seeds new ones.
Alternatives:
Change the key shape (rarely needed; already keyed by Podfile.lock and project file).
Manually purge Actions caches via GitHub UI (org/repo settings), but the version bump is faster and reviewable.
Rule of thumb: only bump when you suspect the cached files don’t reflect your current lockfile/toolchain or you see cache-related flakiness.
